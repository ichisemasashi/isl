# 2.5 ★ 四則電卓を作る（isl-calc 1 周目）

> スパイラル **1 周目** / 基本編 第 5 章 — 題材「isl-calc」を初めて自作する

入門編（0 周目）では、REPL に `(+ 1 (* 2 3))` と打ち込むと `7` が返ってくるのを確かめました。あのとき式を評価していたのは **isl 本体のインタプリタ**です。この章では、その「式を評価する」しくみを **自分の関数として一から作り**ます。これが題材 isl-calc の出発点です。

この章のキーアイデアは一つだけ——**「演算子は関数である」**。ISLISP では `#'+` と書くと「`+` という関数そのもの」を値として取り出せます。これを使うと、電卓の本体がとても短く、しかも拡張しやすくなります。

## 学習目標

この章を終えると、次のことができるようになります。

- 数式を `(op a b ...)` という**リスト（データ）**として受け取れる。
- `#'+` のような**関数参照**で演算子を関数オブジェクトとして取り出せる。
- そのリストを**再帰**でたどり、`apply` で計算する関数 `calc-eval` を書ける。
- `(+ 1 (* 2 3))` のような**入れ子の式**を正しく評価できる。
- 演算子を 1 行足すだけで増やせる、拡張しやすい構造を理解する。

- [2.4 再帰でリストをたどる](04-recursion.md) — 基底ケースと再帰ケース

## 1. アイデア：式は「木」である

`(+ 1 (* 2 3))` をよく見ると、これは次のような**木（ツリー）**です。

```
      (+)
      / \
     1  (*)
        / \
       2   3
```

- 葉（leaf）は **数** … `1`, `2`, `3`
- 節（node）は **演算子つきのリスト** … `(* 2 3)`, `(+ 1 ...)`

評価のルールは、たった 2 つに分けられます。

1. **数（アトム）が来たら、それ自身が答え**。`1` を評価すると `1`。
2. **リストが来たら、先頭が演算子**。残りのオペランドを**先に評価してから**、その演算子で計算する。

ルール 2 の「オペランドを先に評価する」がポイントです。`(* 2 3)` のように、オペランドがまた式（部分木）かもしれないからです。ここで**再帰**が登場します。

---

## 2. 新しい道具：関数参照 `#'` と `apply`

電卓を書く前に、この章で使う 2 つの道具を REPL で確かめます。どちらも「演算子は関数である」という見方を実現するための道具です。

### `#'op` — 関数を「値」として取り出す

ふだん `(+ 1 2)` と書くと `+` は「呼び出され」ますが、`#'+` と書くと **`+` 関数そのもの**を 1 つの値として取り出せます。`#'+` は `(function +)` の短縮記法で、どちらで書いても同じです。

```lisp
> #'+
#<function +>
```

### `funcall` / `apply` — 取り出した関数を呼ぶ

関数オブジェクトは、`funcall`（引数を並べて渡す）か `apply`（引数を**リストで**渡す）で呼び出します。

```lisp
> (funcall #'+ 1 2)        ; 引数を並べて
3
> (apply #'+ '(1 2 3 4))   ; 引数をリストで
10
```

電卓では、オペランドを評価した結果が**リスト**で手に入るので、`apply` がぴったりです。

> **補足**: `#'` ・ `funcall` ・ `apply` は中級編 [3.1 高階関数](../03-intermediate/01-higher-order.md) で改めて体系的に学びます。ここでは「演算子を関数として持ち回る」ための最小限だけ使います。

---

## 3. 演算子を関数に変換する：`op->fn`

「演算子は関数である」を、そのままコードにします。シンボル `+` を関数 `#'+` に対応づける表を、`cond` で書きましょう。

```lisp
(defun op->fn (op)
  (cond ((eq op '+) #'+)
        ((eq op '-) #'-)
        ((eq op '*) #'*)
        ((eq op '/) #'/)
        (t (error "unknown operator" op))))
```

`(eq op '+)` は「`op` がシンボル `+` か？」の判定です（シンボル比較には `eq` を使います）。どれにも当てはまらなければ最後の `t` 節に落ち、エラーになります（`t` は ISLISP で「常に真」を表し、`cond` の既定節に使います）。

```lisp
> (op->fn '*)
#<function *>
```

---

## 4. calc-eval を書く

評価の 2 ルールを `op->fn` と `apply` で組み立てます。

```lisp
(defun calc-eval (e)
  (if (consp e)
      ;; --- ルール2: リスト（式）なら ---
      (let ((fn   (op->fn (car e)))                         ; 演算子 → 関数
            (args (mapcar (lambda (x) (calc-eval x))        ; オペランドを再帰評価
                          (cdr e))))
        (apply fn args))                                    ; 関数をオペランドに適用
      ;; --- ルール1: アトム（数）なら、それ自身 ---
      e))
```

コードを 3 つの観点から読み解きます。

### (1) `consp` で「式かアトムか」を見分ける

`(consp e)` は `e` がコンス（リスト）なら真を返します。真ならルール 2、偽（数など）ならルール 1 です。これが基底ケースと再帰ケースの分かれ目になります。

### (2) `mapcar` でオペランドを「先に」一括評価する

`(cdr e)` は演算子を除いたオペランド列です。`mapcar` で各要素に `calc-eval` を適用し、**部分式をすべて値に変えて**から先へ進みます。`(* 2 3)` のような部分式はここで `6` になります。

### (3) `apply` で関数をオペランドに適用する

`op->fn` が返した関数 `fn` を、評価済みオペランドのリスト `args` に `apply` します。`args` が何個でも `apply` が面倒を見てくれるので、**2 項に限らず可変長**を自然に扱えます。

## 5. 動かして確かめる

上の定義をファイルに保存し、確認用のコードを足します。本リポジトリには動く版を [`code/isl-calc-v1.lsp`](code/isl-calc-v1.lsp) として同梱しています。

```lisp
(format (standard-output) "~S => ~S~%" '(+ 1 2)       (calc-eval '(+ 1 2)))
(format (standard-output) "~S => ~S~%" '(+ 1 (* 2 3)) (calc-eval '(+ 1 (* 2 3))))
(format (standard-output) "~S => ~S~%" '(+ 1 2 3 4)   (calc-eval '(+ 1 2 3 4)))
(format (standard-output) "~S => ~S~%" '(- 5)         (calc-eval '(- 5)))
(format (standard-output) "~S => ~S~%" '(/ 7 2)       (calc-eval '(/ 7 2)))
```

実行します。

```sh
./bin/isl docs/textBook/part1-core/02-basic/code/isl-calc-v1.lsp
```

出力:

```
(+ 1 2) => 3
(+ 1 (* 2 3)) => 7
(+ 1 2 3 4) => 10
(- 5) => -5
(/ 7 2) => 7/2
```

入れ子の `(+ 1 (* 2 3))` が `7` になり、自作の電卓が完成しました 🎉 しかも `apply` のおかげで、**可変長の `(+ 1 2 3 4)`** も **単項マイナス `(- 5)`** も、特別扱いせずにそのまま動いています。

> **ISLISP の数のはなし**: `(/ 7 2)` が `3` でも `3.5` でもなく **`7/2`（有理数）** になった点に注目してください。ISLISP は整数同士の割り算を、誤差のない**正確な有理数**で表します。小数が欲しいときは `(/ 7.0 2)` のように浮動小数点数を使います。数値の扱いは標準編でさらに掘り下げます。

## 6. エラーも試す

未知の演算子 `%` を渡すと、`op->fn` の `t` 節が発火します。

```lisp
> (calc-eval '(% 1 2))
*** ERROR: unknown operator (%)
```

数値計算が破綻する前に、自分の言葉でエラーを出せました。エラーを「投げる」だけでなく「**捕まえる**」方法は、中級編の [3.4 エラー処理の基礎](../03-intermediate/04-error-handling-intro.md) で学びます。

## 7. この設計のうれしさ — 拡張は 1 行

「演算子は関数である」という設計の利点は、**演算子を増やすのが簡単**なことです。`calc-eval` 本体には一切手を入れず、`op->fn` の表に 1 行足すだけで済みます。

```lisp
(defun op->fn (op)
  (cond ((eq op '+) #'+)
        ((eq op '-) #'-)
        ((eq op '*) #'*)
        ((eq op '/) #'/)
        ((eq op 'max) #'max)   ; ← 1行足すだけ
        ((eq op 'min) #'min)   ; ← 1行足すだけ
        (t (error "unknown operator" op))))
```

```lisp
> (calc-eval '(max 1 (+ 4 5) 3))
9
```

演算子と実装が `op->fn` の表に集約されているおかげで、拡張点が一目でわかります。比較演算などを足した版は [`code/isl-calc-v1-ext.lsp`](code/isl-calc-v1-ext.lsp) にあります。

## 8. この電卓の「限界」を見ておく

便利になった一方で、はっきりした限界も残っています。これは欠点ではなく、**次の周で何を足すかの予告**です。

- **変数が無い** … `(+ x 1)` の `x` を解決できない（`consp` でないシンボルはそのまま返ってしまう）。
- **ユーザ定義関数が無い** … 使えるのは `op->fn` に並べた組み込み演算だけ。
- **構造が平たい** … 演算子ごとの違いが `op->fn` の表に集約されているのは良いが、`let` のような「特殊な評価の仕方をするフォーム」（引数を先に評価しては困るもの）は扱えない。

2 周目（中級編）では、この `cond` の表を **オブジェクトシステムの AST ノード＋ジェネリック関数**へ作り直し、**変数束縛**と**ユーザ定義関数**を加えて、これらの限界を一つずつ越えていきます。

1. **比較演算**: `(> a b)` と `(< a b)` を足せ。真なら `t`、偽なら `nil` が返ることを確認せよ。
   ヒント: `op->fn` に `#'>` と `#'<` を足すだけ。`calc-eval` は変更不要。

## まとめ

- 数式は `(op a b ...)` という**リスト＝データ**であり、評価器はそれを**木としてたどる関数**である。
- ルールは 2 つだけ：**アトムはそれ自身**、**リストは「オペランドを再帰評価してから演算」**。
- ISLISP の **`#'op`（関数参照）**で演算子を関数として取り出し、**`apply`** で適用する——この一手で可変長引数も単項マイナスも自然に扱える。
- 演算子と実装を `op->fn` に集約したので、拡張は 1 行で済む。
- いまの限界（変数・ユーザ定義関数）が、次の周でオブジェクトシステムを導入する動機になる。
