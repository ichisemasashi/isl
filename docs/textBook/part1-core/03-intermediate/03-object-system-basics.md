# 3.3 オブジェクトシステムの基礎

> スパイラル **2 周目** / 中級編 第 3 章

基本編の電卓は、演算子の種類を `op->fn` の中の `cond` で分岐していました。「もし `+` なら…、もし `-` なら…」という**分岐の羅列**です。式の種類が増える（数・変数・`let`・関数呼び出し…）と、この `cond` はどんどん長くなり、しかも評価・表示・最適化と処理を増やすたびに**似た `cond` を何度も書く**ことになります。

ISLISP には、**クラス**と**総称関数（ジェネリック関数）**からなる**オブジェクトシステム**が規格として備わっています。これは、この「型で分岐する」を言語のしくみに引き上げる道具です。**データに型（クラス）を与え、処理を型ごとのメソッドに分けて書く**——分岐は処理系が肩代わりします。この章で道具立てを学び、5 章で電卓の `cond` をこのオブジェクトシステムに置き換えます。

> **用語の注意（CLOS との関係）**: ISLISP のオブジェクトシステムは、Common Lisp の **CLOS（Common Lisp Object System）由来**のサブセットで、基本形の `defclass` / `defgeneric` / `defmethod` は共通します。ただし CLOS そのものではありません——インスタンス生成が `make-instance` ではなく **`create`** である、フル MOP を持たない、などの違いがあります。そして重要な点として、このオブジェクトシステムは **ISLISP 規格（strict プロファイル）に含まれる標準機能**です。「CLOS」は本来 Common Lisp の呼称なので、本書では以降、単に「オブジェクトシステム」と呼びます。

## 学習目標

この章を終えると、次のことができるようになります。

- `defclass` でクラスとスロット（フィールド）を定義できる。
- `create` でインスタンスを作り、アクセサでスロットを読める。
- `defgeneric` / `defmethod` でジェネリック関数を定義し、**型でディスパッチ**できる。
- 電卓の `cond` 分岐を「ノード型ごとのメソッド」に置き換える発想を持てる。

## 1. クラスとスロット——`defclass`

クラスは「同じ種類のデータの設計図」です。まず、電卓の AST（抽象構文木）で使う**数ノード**を定義します。数ノードは値を 1 つ持てばよいので、`value` というスロット（フィールド）を 1 つ与えます。

```lisp
> (defclass num-node ()
    ((value :initarg value :accessor node-value)))
num-node
```

読み下します。

- `num-node` … クラス名。
- `()` … 親クラスのリスト（今回は無し）。
- `((value ...))` … **スロット定義**のリスト。ここでは `value` が 1 つ。
- `:initarg value` … インスタンス生成時に、この値を渡すための**引数名**を `value` とする。
- `:accessor node-value` … スロットを読み書きする**アクセサ関数** `node-value` を自動生成する。

### インスタンスを作る——`create`

クラスから実体（インスタンス）を作るには `create` を使います。`(class クラス名)` でクラスオブジェクトを取り出し、`:initarg` で決めた引数名と値を並べます。

```lisp
> (defglobal n (create (class num-node) 'value 42))
n
> (node-value n)
42
```

`(create (class num-node) 'value 42)` は「`num-node` のインスタンスを、`value` を `42` にして作れ」という意味です。作ったインスタンスのスロットは、アクセサ `node-value` で読み出せます。

---

## 2. スロットを 2 つ持つクラス

演算ノードは、演算子と左右のオペランドを持ちます。スロットを複数並べるだけです。

```lisp
> (defclass binop-node ()
    ((op  :initarg op  :accessor node-op)
     (lhs :initarg lhs :accessor node-lhs)
     (rhs :initarg rhs :accessor node-rhs)))
binop-node
```

木を組み立てます。ポイントは、`lhs`・`rhs` に**入れるのも AST ノード**だということです。`(+ 1 2)` の `1` と `2` も `num-node` にして、演算ノードの子にします。

```lisp
> (defglobal one (create (class num-node) 'value 1))
one
> (defglobal two (create (class num-node) 'value 2))
two
> (defglobal expr (create (class binop-node) 'op '+ 'lhs one 'rhs two))
expr
> (node-op expr)
+
```

これで `(+ 1 2)` を「`op=+`、左右に `num-node` をぶら下げた `binop-node`」という**木**で表せました。あとは「この木を評価する」処理を書くだけです。

---

## 3. 型でディスパッチする——`defgeneric` と `defmethod`

いよいよオブジェクトシステムの核心です。「`num-node` なら値をそのまま、`binop-node` なら左右を評価して演算」という処理を、**型ごとに別々のメソッド**として書きます。

まず、名前だけの**ジェネリック関数**を宣言します。

```lisp
> (defgeneric ast-eval (node))
ast-eval
```

`defgeneric` は「`ast-eval` という関数を、これから型ごとに定義していく」という宣言です。次に、型ごとの実装（**メソッド**）を `defmethod` で足します。引数リストの `(n num-node)` が「`n` が `num-node` 型のときに、このメソッドを使え」という**ディスパッチ指定**です。

```lisp
> (defmethod ast-eval ((n num-node))
    (node-value n))
ast-eval
> (defmethod ast-eval ((n binop-node))
    (funcall (cond ((eq (node-op n) '+) #'+)
                   ((eq (node-op n) '-) #'-)
                   ((eq (node-op n) '*) #'*)
                   ((eq (node-op n) '/) #'/))
             (ast-eval (node-lhs n))     ; 左を再帰評価
             (ast-eval (node-rhs n))))   ; 右を再帰評価
ast-eval
```

呼び出してみます。`ast-eval` を呼ぶと、渡した**インスタンスの型に応じて**、適切なメソッドが自動で選ばれます。

```lisp
> (ast-eval one)      ; one は num-node → 数のメソッド
1
> (ast-eval expr)     ; expr は binop-node → 演算のメソッド
3
```

`(ast-eval expr)` の中で `(ast-eval (node-lhs n))` が呼ばれると、その引数（`one`）はまた `num-node` なので、**さらに型で分岐**して数のメソッドに落ちます。1 周目で `calc-eval` が再帰したのと同じ木のたどり方が、ここでは「メソッドの再帰呼び出し＋型ディスパッチ」で実現されています。子がまた `binop-node` なら、そこも同じように評価されます。

## 4. `cond` からオブジェクトシステムへ——何が変わったか

1 周目の `cond` 版と、このオブジェクトシステム版を並べてみます。

| | 1 周目（`cond`） | 2 周目（オブジェクトシステム） |
|---|---|---|
| 式の種類 | `consp` かどうかで `if` 分岐 | ノードの**クラス**で自動ディスパッチ |
| 処理の追加 | `calc-eval` 本体を書き換える | 新しい `defmethod` を**足す**だけ |
| 新しい型の追加 | `if`/`cond` の枝を増やす | 新しい `defclass` ＋メソッドを足す |

いちばんの違いは**拡張の仕方**です。`cond` 版は分岐の中心を触り続けますが、オブジェクトシステム版は「新しい型」も「新しい処理」も、既存コードに手を入れず**追加**で対応できます。式の種類（数・変数・`let`・関数呼び出し…）が増えていく電卓にとって、この性質は効いてきます。

> **メソッド結合について（軽く）**: 同じジェネリック関数に複数のメソッドがあるとき、どれが選ばれ、どう組み合わされるか（`:before` / `:after` / 継承と `next-method` など）を**メソッド結合**と呼びます。ここでは「型がいちばん合うメソッドが選ばれる」ことだけ押さえれば十分です。深入りは上級編に譲ります。

---

## ★ 処理系の中では（コラム）

`core.scm` はオブジェクトシステムを、クラス表 `*class-table*` とジェネリック関数の**メソッド表**で管理しています。`ast-eval` を呼ぶと、処理系は引数の実際のクラスを調べ、そのクラス（と親クラス）に最も適合するメソッドを選んで呼び出します。私たちが手で書いていた `cond` の型分岐を、処理系がメソッド表の探索として**内部で**行っている、と考えると腑に落ちます。アクセサ（`node-value` など）も、スロット位置を引く小さな関数として自動生成されています。詳しくは 6 周目（内部編）で。

## 演習

1. **加算ノード**: `add-node`（`lhs` と `rhs` の 2 スロット）を `defclass` で定義し、`create` でインスタンスを作って両アクセサで読み出せ。
2. **専用メソッド**: 上の `add-node` に対する `(defmethod ast-eval ((n add-node)) ...)` を書き、左右を足して返せ。`binop-node` と違い、演算子を持たず必ず加算する版でよい。
3. **表示メソッド**: `ast-eval` とは別に `(defgeneric ast-show (node))` を宣言し、`num-node` は値を、`binop-node` は `(op lhs rhs)` らしき文字列を返すメソッドを書け（評価と表示を**別のジェネリック関数**に分けられることを体験する）。

---

## まとめ

- `defclass` で**クラスとスロット**を定義し、`:initarg`（生成時の引数名）と `:accessor`（読み書き関数）を与える。`create` でインスタンスを作る。
- `defgeneric` でジェネリック関数を宣言し、`defmethod` で**型ごとのメソッド**を足す。呼び出し時は引数の型で自動ディスパッチされる。
- これは電卓の `cond` 型分岐を言語機構に置き換えたもの。**新しい型も新しい処理も「追加」で対応**できるのがオブジェクトシステムの強み。
- メソッドの再帰呼び出し＋型ディスパッチで、木のたどり方（＝評価）が自然に書ける。
