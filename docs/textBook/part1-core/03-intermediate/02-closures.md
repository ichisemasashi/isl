# 3.2 クロージャ

> スパイラル **2 周目** / 中級編 第 2 章

前章の最後で、`make-adder` が「`n` を覚えた関数」を返すのを見ました。関数が、作られたときの変数を**覚えている**——このしくみが**クロージャ**（closure）です。この章では、なぜ覚えていられるのか（レキシカルスコープ）を理解し、クロージャで「状態」を持ち回れることを体験します。

そしてこの章には、スパイラル上の重要な役割があります。クロージャや連想リストで表す**「名前 → 値」の対応表＝環境**が、5 章で作り直す「**変数を持つ電卓**」の心臓部になるのです。

## 学習目標

この章を終えると、次のことができるようになります。

- レキシカルスコープと**自由変数の捕捉**を説明できる。
- クロージャで**状態**（隠された変数）を持ち回れる。
- 「名前 → 値」の対応表を**連想リスト**で表し、`lookup` を書ける。

## 前提知識

- [3.1 高階関数](01-higher-order.md) — 関数を返す `make-adder`

## 1. レキシカルスコープ——名前は「書かれた場所」で決まる

もう一度 `make-adder` を見ます。

```lisp
> (defun make-adder (n)
    (lambda (x) (+ x n)))
make-adder
```

`(lambda (x) (+ x n))` の中に `n` が出てきますが、`n` はこの `lambda` の引数ではありません。`lambda` を**囲んでいる** `make-adder` の引数です。このように、自分の引数でない変数を**自由変数**（free variable）と呼びます。

ISLISP は**レキシカルスコープ**を採ります。これは「変数がどれを指すかは、コードが**書かれた場所**で決まる」という規則です。`lambda` の中の `n` は、その `lambda` が書かれた場所——つまり `make-adder` の `n`——を指します。`lambda` が作られると、この `n` の束縛が関数といっしょに**閉じ込められ**ます。これが「クロージャ（閉包）」の名の由来です。

```lisp
> (defglobal add10 (make-adder 10))   ; n=10 を閉じ込めた関数
add10
> (funcall add10 5)
15
> (funcall (make-adder 100) 5)        ; こちらは n=100 を閉じ込めた別の関数
105
```

`add10` と `(make-adder 100)` は同じ `lambda` から生まれていますが、閉じ込めた `n` が違うので別々に振る舞います。**同じコードでも、捕捉した環境が違えば別の関数**なのです。

---

## 2. クロージャは「状態」を隠し持てる

捕捉した変数を `setq` で書き換えれば、呼ぶたびに変化する関数——つまり**状態を持つ関数**が作れます。定番のカウンタを書いてみます。

```lisp
> (defun make-counter ()
    (let ((count 0))
      (lambda ()
        (setq count (+ count 1))
        count)))
make-counter
> (defglobal c (make-counter))
c
> (funcall c)
1
> (funcall c)
2
> (funcall c)
3
```

`count` は `make-counter` の中の `let` でだけ見える局所変数です。外から `count` に触る方法はありません。それでも `c` を呼ぶたびに `1, 2, 3` と増えていく——`c` が `count` の束縛を捕捉し、内部に隠し持っているからです。

別のカウンタを作れば、状態は**独立**しています。

```lisp
> (defglobal d (make-counter))
d
> (funcall d)      ; d は自分の count を持つ
1
> (funcall c)      ; c の count は 3 の続き
4
```

「変数を外から隠しつつ、関数だけを公開する」——クロージャはオブジェクト指向の**カプセル化**に近いことを、たった数行で実現します。

## 3. 環境を「データ」で表す——連想リスト

クロージャは「名前 → 値」の対応を**言語のしくみ**（束縛）で持ちます。一方、電卓を作るときに必要なのは、それを**自分のデータ**として持ち、自由に足したり引いたりできる形です。もっとも手軽なのが**連想リスト**（association list, alist）です。

連想リストは `(名前 . 値)` のペアを並べたリストです。

```lisp
> (defglobal env '((x . 10) (y . 20)))
env
> (assoc 'x env)
(x . 10)
> (cdr (assoc 'x env))
10
```

`assoc` は「キーで探して、見つかったペアを返す」関数です。値そのものが欲しければ `cdr` を取ります。これを関数にまとめると、電卓の**変数参照**がそのまま書けます。

```lisp
> (defun lookup (name env)
    (let ((pair (assoc name env)))
      (if pair
          (cdr pair)
          (error "unbound variable" name))))
lookup
> (lookup 'y env)
20
> (lookup 'z env)
*** ERROR: unbound variable (z)
```

見つからなければ「未定義変数」エラーを出す——ここまで来ると、電卓に変数を持たせる準備がほぼ整っています。

### 環境を「広げる」——束縛の追加

`let` のような束縛を評価するとき、電卓は**既存の環境に新しいペアを足した環境**を作ります。これは `cons` 一発です。

```lisp
> (defglobal env2 (cons (cons 'z 30) env))   ; z=30 を先頭に足す
env2
> (lookup 'z env2)
30
> (lookup 'x env2)     ; もとの束縛もそのまま見える
10
```

**古い環境を壊さず、先頭に足した新しい環境を作る**。この「継ぎ足し」が、5 章の `let-node` の評価そのものになります。先頭に足すことで、同じ名前があれば新しい束縛が**先に**見つかる（＝内側のスコープが優先される）点も、ちょうど都合よく働きます。

---

## ★ 処理系の中では（コラム）

`core.scm` の評価器は、変数の束縛を**フレーム**（`vector`）で表し、参照時には内側のフレームから外側へ順にたどって名前を探します（`frame-find-pair`）。私たちがこの章で連想リストと `assoc` を使って手作りした「環境をたどって名前を引く」しくみを、本物の処理系はより効率のよいデータ構造で行っている——というだけの違いです。クロージャは、この「捕捉したフレームへの参照」を関数オブジェクトに持たせることで実現されています。6 周目（内部編）で `core.scm` を開いて確認します。

## 演習

1. **連想リストの `lookup`**: 本文の `lookup` を自分で打ち込み、`'((a . 1) (b . 2))` から `a` と `b` を引け。存在しないキーでエラーになることも確かめよ。
2. **環境の継ぎ足し**: `env` に `(w . 99)` を足した `env3` を `cons` で作り、`lookup` で `w` と、もとからある `x` の両方が引けることを確かめよ。
3. **カウンタ（発展）**: `make-counter` を改造し、増分を引数で受け取る `make-stepper`（`(make-stepper 5)` は呼ぶたびに 5 ずつ増える）を書け。

## まとめ

- ISLISP は**レキシカルスコープ**。`lambda` 中の自由変数は「書かれた場所」の束縛を指し、それが関数と一緒に**捕捉**される＝クロージャ。
- クロージャは捕捉した変数を隠し持てるので、**状態を持つ関数**（カウンタ）が書ける。状態はクロージャごとに独立する。
- 電卓の**環境**は、`(名前 . 値)` の**連想リスト**で表せる。`assoc`＋`cdr` で参照（`lookup`）、`cons` で継ぎ足し（束縛の追加）。
- この「環境をたどって名前を引く／先頭に足す」しくみが、5 章の変数付き電卓の心臓部になる。
