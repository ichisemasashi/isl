# C1.1 islc 概観

> スパイラル **5 周目** / コンパイラ編（part2 第 3 編）第 1 章

ここまで、isl-calc は一貫して **`./bin/isl`（インタプリタ）**で動かしてきました。インタプリタは、プログラムを読みながら**その場で解釈して実行**します。手軽で対話的ですが、同じ式を何度も評価するときは、毎回「解釈」のコストを払います。

5 周目のテーマは、その解釈コストを**前もって払っておく**——プログラムを**機械語（ネイティブコード）にコンパイル**する道具、**`islc`** です。isl には LLVM をバックエンドにしたコンパイラが備わっており、ISLISP プログラムをネイティブバイナリにしたり、即時コンパイルして実行したりできます。この編では islc の全体像をつかみ、最後に isl-calc をネイティブ化して速度差を体感します。

## 学習目標

この章を終えると、次のことができるようになります。

- インタプリタとコンパイラの違いを、isl の文脈で説明できる。
- コンパイルのパイプライン（read → IR → LLVM → ネイティブ）の全体像を描ける。
- `islc` の 4 つのモード（emit-llvm / run / AOT / jit）の役割を使い分けられる。

## 前提知識

- part1-core を読了。できれば標準編（3 周目）も。

## 1. インタプリタとコンパイラ

同じプログラムを実行するのに、二つのやり方があります。

- **インタプリタ**（`./bin/isl`）… プログラムを読みながら、式を一つずつ解釈して実行する。準備が要らず対話に向くが、実行のたびに解釈コストがかかる。
- **コンパイラ**（`./bin/islc`）… プログラムを先に**機械語へ翻訳**しておき、あとはその機械語を CPU が直接実行する。翻訳のひと手間がかかるが、実行は速い。

料理にたとえるなら、インタプリタは「レシピを一行読んでは作り、また一行読んでは作る」やり方、コンパイラは「レシピを全部先に段取りして、あとは一気に作る」やり方です。**同じ計算を大量に繰り返す**ほど、後者の効き目が出ます。5 周目の最後で、この差を実測します。

---

## 2. コンパイルのパイプライン

islc は、ISLISP のソースを一気に機械語にするのではなく、いくつかの段階を踏みます。

```
ソース(.lsp)
   │  read           … S式として読む
   ▼
S式
   │  macroexpand    … マクロを展開する
   ▼
展開済み S式
   │  IR 正規化       … 中間表現(IR)へ（frontend.scm）
   ▼
IR
   │  lowering        … LLVM 向けの直交 CFG へ（基本ブロック分解）
   ▼
LLVM IR (.ll)
   │  clang           … ネイティブコードへ
   ▼
ネイティブバイナリ
```

途中に **IR（中間表現）** という「S式でも機械語でもない、翻訳しやすい形」を挟むのがポイントです。なぜわざわざ挟むのかは次章 [C1.2](02-understanding-ir.md) で扱います。最終段では **clang**（LLVM のフロントエンド）を呼び、LLVM IR からネイティブバイナリを生成します。

---

## 3. islc の 4 つのモード

islc は、パイプラインのどこまで進めるか・生成物を残すかで、いくつかのモードを持ちます。すべて `examples/hello.lsp`（`10` の階乗を print する小さなプログラム）で試せます。

### `--emit-llvm` — LLVM IR を出力する

パイプラインを LLVM IR まで進め、テキストファイルに書き出します。中身を覗きたいときに使います（[C1.3](03-aot-build.md)）。

```sh
./bin/islc --emit-llvm out.ll examples/hello.lsp
```

### `--run` — その場でコンパイルして実行する

バイナリを残さず、コンパイルして即実行します。手軽に「コンパイルすると動くか」を確かめられます。

```sh
./bin/islc --run examples/hello.lsp
# => 3628800
```

> 実行時に `overriding the module target triple` という警告が出ることがありますが、これは LLVM にターゲット CPU を明示している旨の**無害な通知**です。結果には影響しません。

### `islc-aot` — 配布用のネイティブバイナリを作る

ワンステップでネイティブバイナリを生成します。生成物は単体で実行できる普通の実行ファイルです（[C1.3](03-aot-build.md)）。

```sh
./bin/islc-aot -o hello-aot examples/hello.lsp
./hello-aot
# => 3628800
```

### `--jit` — 即時コンパイル実行（JIT）

実行の直前に機械語へコンパイルし、そのまま走らせます。バイナリを残さない点は `--run` に似ていますが、内部は ORC v2 という JIT エンジンです（[C1.4](04-jit-run.md)）。

```sh
./bin/islc --jit examples/hello.lsp
# => 3628800
```

まとめると、**中身を見る**なら `--emit-llvm`、**手軽に試す**なら `--run`、**配る**なら `islc-aot`、**即時コンパイルで走らせる**なら `--jit`、という役割分担です。

## 4. マイルストーン M0〜M11

isl のコンパイラは一度に作られたのではなく、**M0 から M11 までの段階（マイルストーン）**を積み上げて構築されました。数値・関数・分岐・クロージャ・末尾呼び出し・そして JIT（M11）と、扱える機能が段階的に増えています。裏返せば、**コンパイラが扱える ISLISP の範囲には「ここまで」という境界がある**ということです。この境界は 5 周目の capstone（[C1.5](05-calc-native-binary.md)）で実際にぶつかることになります——そしてそれは失敗ではなく、コンパイラを理解するうえで大切な学びです。

各マイルストーンの定義は `docs/compiler-milestones.md` にまとまっています。

---

## ★ 処理系の中では（コラム）

`./bin/islc` は Gauche で書かれたコンパイラドライバで、`src/isl/compiler/` 以下のモジュール（`frontend.scm` / `ir.scm` / `lowering.scm` / `codegen.scm` / `runner.scm` / `jit.scm`）を束ねています。`--run` も `islc-aot` も、途中までは同じパイプラインを通り、最後に「一時バイナリを実行して捨てる」か「バイナリを残す」かが違うだけです。生成されたネイティブコードは、`isl_rt_*` という**ランタイム関数**（数の生成・変数参照・関数適用など）を呼び出しながら動きます。その姿は次章以降で少しずつ覗きます。

## まとめ

- **インタプリタ**は解釈しながら実行、**コンパイラ**は先に機械語へ翻訳してから実行。繰り返しが多いほどコンパイルが効く。
- islc のパイプライン: read → macroexpand → **IR** → LLVM IR → clang → ネイティブ。
- 4 モード: `--emit-llvm`（中身を見る）/ `--run`（手軽に試す）/ `islc-aot`（配る）/ `--jit`（即時コンパイル）。
- コンパイラは M0〜M11 で構築され、扱える範囲に境界がある（C1.5 で体感）。
