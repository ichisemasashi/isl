# C1.5 ★ ネイティブバイナリにする（isl-calc 5 周目）

> スパイラル **5 周目** / コンパイラ編 第 5 章 — isl-calc を高速化する

5 周目の締めくくりです。ここまでで islc の 4 モードと IR を学びました。いよいよ isl-calc を **ネイティブコンパイル**し、インタプリタとの速度差を実測します。

ただしこの周には、他の周と違う「予想外の壁」があります。**3 周目で作った strict 版 v3（オブジェクトシステム＋コンディション）は、そのままではコンパイルできません**。この壁にぶつかること自体が、コンパイラを理解する最良の教材になります。壁を確かめ、それを避けて「コンパイルできる isl-calc」を作り、速度を測る——それがこの章の道のりです。完成版は [`code/isl-calc-bench.lsp`](code/isl-calc-bench.lsp) に同梱しています。

## 学習目標

この章を終えると、次のことができるようになります。

- コンパイラの「守備範囲（コンパイル可能なサブセット）」を、実物で確かめられる。
- コンパイル可能な calc を用意し、`islc-aot` でネイティブバイナリにできる。
- インタプリタ・AOT・JIT の速度を実測し、差の理由を説明できる。

## 1. 壁にぶつかる——v3 はコンパイルできない

まず、3 周目の到達点である strict 版 v3 をコンパイルしようとしてみます。

```sh
./bin/islc --run docs/textBook/part2-tracks/01-standard/code/isl-calc-v3.lsp
```

```
isl_rt: native backend: unsupported special form: class
```

コンパイラが「`class` という特殊形式はサポートしていない」と告げて止まりました。v3 は **オブジェクトシステム**（`defclass` / `create` / `defgeneric`）と **`with-handler`** の両方に依存しており、どちらも native backend の守備範囲の**外**です（[C1.1](01-islc-overview.md) の M0〜M11 で触れた「境界」）。

> 名指しされる特殊形式は、コンパイラが先に到達した方（`class` または `with-handler`）になります。どちらのメッセージでも意味は同じ——「v3 が使っている仕組みは、まだコンパイルできない」ということです。

これは欠陥ではなく、コンパイラ設計の**現実**です。豊かな言語機能をすべて機械語に落とすのは大仕事で、コンパイラはまず「よく使う芯」から対応していきます。裏を返せば——**コンパイルしたいなら、その芯の範囲で書く**必要がある、ということです。

---

## 2. 壁を避ける——1 周目のリスト評価器に戻る

では、何ならコンパイルできるのか。答えは、**1 周目のリスト評価器**です。あの素朴な `cond` + `apply` + `mapcar` + 再帰は、ちょうどコンパイラの守備範囲に収まります。

```lisp
(defun op->fn (op)
  (cond ((eq op '+) #'+) ((eq op '-) #'-)
        ((eq op '*) #'*) ((eq op '/) #'/)
        (t (error "unknown operator"))))

(defun calc-eval (e)
  (if (consp e)
      (apply (op->fn (car e))
             (mapcar (lambda (x) (calc-eval x)) (cdr e)))
      e))
```

`--run` で確かめます。

```sh
./bin/islc --run（このコードを含むファイル）
# => 7   （(+ 1 (* 2 3)) を評価した場合）
```

今度は通りました。**周を重ねて豊かにしてきた isl-calc を、あえて素朴な姿に戻す**——遠回りに見えますが、これは「コンパイルという制約の下で、何を諦め何を保つか」を考える設計の訓練です。オブジェクトシステムの見通しの良さは失いますが、四則計算の本質は 1 周目のままちゃんと動きます。

---

## 3. ベンチマークにする——固定式を大量評価

コンパイルの効き目は「同じ計算を大量に繰り返す」ときに出ます。そこで、固定した式を何度も評価する**ベンチマーク**にします。

```lisp
(defglobal expr '(+ 1 (* 2 (- 100 (/ 40 4)))))
(defglobal acc 0)
(dotimes (i 3000)
  (setq acc (calc-eval expr)))
(print acc)   ; => 181
```

> **なぜ標準入力から式を読まないのか**: 対話的な電卓なら「標準入力から式を読む」形にしたいところですが、現状の native backend はコンパイル版での標準入力 `read` に対応していません。そこで式は**プログラムに埋め込み**ます。また、反復数を極端に大きくするとネイティブランタイムが不安定になることがあるため、ここでは 3000 回に抑えています。「コンパイルできる範囲・安定して動く範囲を見極める」のも、この周の学びの一部です。

## 4. 速度を測る——インタプリタ vs AOT vs JIT

同じ [`isl-calc-bench.lsp`](code/isl-calc-bench.lsp) を、3 つの方法で走らせて時間を比べます。

```sh
# インタプリタ
time ./bin/isl        docs/textBook/part2-tracks/03-compiler/code/isl-calc-bench.lsp

# AOT（ビルドして実行）
./bin/islc-aot -o isl-calc-bench docs/textBook/part2-tracks/03-compiler/code/isl-calc-bench.lsp
time ./isl-calc-bench

# JIT
time ./bin/islc --jit docs/textBook/part2-tracks/03-compiler/code/isl-calc-bench.lsp
```

手元の環境（macOS / arm64）での実測は、およそ次のようになりました（環境で変わります）。

| 実行方法 | おおよその実行時間 | 備考 |
|---------|-----------------|------|
| インタプリタ `./bin/isl` | **約 8 秒** | 毎回「解釈」しながら評価 |
| AOT バイナリ | **約 0.1 秒** | もう機械語。ほぼ計算時間だけ |
| JIT `--jit` | 数秒（コンパイル込み） | 実行前にまとめてコンパイル |

AOT はインタプリタの **およそ 100 倍**速く終わりました。同じ `calc-eval`・同じ 3000 回の評価でも、「毎回解釈する」か「一度機械語にして走る」かで、これだけ差が出ます。5 周目の狙い——**コンパイルの効き目を体感する**——が数字で確かめられました。

JIT がこの小さな題材で速く見えないのは、[C1.4](04-jit-run.md) で触れたとおり、**実行前にコンパイル費用を前払い**しているからです。「一度ビルドして繰り返し使う」なら AOT が、この用途では有利だとわかります。

## 5. コンパイル可能性の境界（まとめの地図）

この周でぶつかった「境界」を整理しておきます。

| 機能 | コンパイル | 備考 |
|------|:--:|------|
| 算術・`cond`・`if`・再帰 | ✅ | 芯。速くできる |
| `apply` / `mapcar` / `funcall` / `#'op` | ✅ | 1 周目の評価器が動く |
| `dotimes` / `defglobal` / `setq` | ✅ | ベンチに使用 |
| オブジェクトシステム（`defclass`/`create`） | ❌ | `unsupported special form: class` |
| `with-handler`（コンディション） | ❌ | 未サポート |
| 標準入力 `read` | ❌ | コンパイル版では不可 |
| 拡張 I/O・ネットワーク（4 周目の機能） | ❌ | ランタイム外 |

「速くしたい芯」はコンパイルでき、「表現力を上げる仕組み（オブジェクトシステム・コンディション）」や「外界とのやり取り（I/O・ネット）」はインタプリタの領分——という住み分けが見えてきます。実用では、**芯をコンパイルし、周辺はインタプリタで**という組み合わせも考えられます。

---

## ★ 処理系の中では（コラム）

native backend が `class` や `with-handler` を「unsupported」と言うのは、コンパイラのフロントエンド／lowering がこれらの特殊形式に対応する変換規則をまだ持たないためです（M0〜M11 で対応した範囲の外）。一方でインタプリタ `core.scm` は全機能を持ちます。だからこそ本書のスパイラルは、豊かな機能はインタプリタで学び（2〜4 周目）、速度が要る芯だけをコンパイルする（5 周目）、という順路になっています。「何でもコンパイルできるわけではない」と知ることは、コンパイラを過信も過小評価もしない、健全な理解につながります。

## 演習

1. **境界の確認**: v3（[`isl-calc-v3.lsp`](../01-standard/code/isl-calc-v3.lsp)）を `./bin/islc --run` し、どの特殊形式で止まるかメッセージを読め。次に 1 周目相当のリスト版が `--run` で通ることを確かめ、両者の差を一言で述べよ。
2. **AOT と JIT の実測**: [`isl-calc-bench.lsp`](code/isl-calc-bench.lsp) をインタプリタ・AOT・JIT で `time` 計測し、表にせよ。AOT が最速になる理由と、JIT がこの題材で不利な理由を書け。
3. **式を変える**: ベンチの `expr` をもっと深い入れ子（例: 二重・三重にネスト）に変え、インタプリタと AOT の差がどう変わるか観察せよ。反復数を上げすぎると不安定になる点にも注意せよ。

## まとめ

- strict 版 v3 は **オブジェクトシステム・`with-handler` 依存でコンパイル不可**。これがコンパイラの守備範囲を教えてくれる。
- コンパイルできる芯（1 周目のリスト評価器）に戻し、固定式を大量評価する**ベンチマーク**にする。
- 実測で AOT はインタプリタの**およそ 100 倍**高速。JIT はコンパイル費用を前払いするため小題材では不利。
- コンパイル可能性には**境界**がある。芯は速く、表現力・外界は interpreter の領分。
