# S1.3 シンボル解決と名前設計

> スパイラル **3 周目** / 標準編 第 3 章

isl-calc は周を重ねるごとに関数が増えてきました。`parse`・`ast-eval`・`op->fn`・`env-lookup`…。プログラムが大きくなると避けられないのが**名前の衝突**です。自分の `parse` と、ライブラリの `parse` がぶつかったら？ この章では、名前をどう管理するか——**シンボル解決**の考え方と、strict で移植性を保つための**名前設計**を扱います。

あわせて、Common Lisp 由来の**パッケージ**機能について、ISLISP 規格上の位置づけをはっきりさせます。結論を先に言うと、strict を意識するコードでは**パッケージに頼らない**のが安全です。その理由と、代わりに使う名前設計を学びます。

## 学習目標

この章を終えると、次のことができるようになります。

- 名前空間が必要になる理由（名前衝突）を説明できる。
- シンボルがどう解決されるか、その大枠をつかむ。
- パッケージが ISLISP 規格外（拡張分類）であることを理解し、strict で非依存に保てる。
- 接頭辞などの規律で、公開 API の名前を衝突しにくく設計できる。

- [S1.2 標準コンディション機構](02-standard-conditions.md)

## 1. なぜ名前空間が要るのか

プログラムの中で `parse` という名前を定義すると、それはグローバルな名前表に載ります。別のモジュールも `parse` を定義したら、後勝ちで上書きされるか、衝突します。小さなうちは気になりませんが、コードが育ち、他人のコードと組み合わせるようになると、**名前をどう仕切るか**が品質を左右します。

この「名前を仕切る箱」が**名前空間**（namespace）です。多くの Lisp では、名前空間の実装として**パッケージ**が使われます。ただし——後述するように、ISLISP 規格ではこの箱の扱いが Common Lisp とは異なります。

---

## 2. シンボル解決の大枠

私たちがコードに書く `parse` という**名前**（シンボル）は、評価時に「どの束縛を指すか」へと結び付けられます。これを**シンボル解決**と呼びます。isl の解決は、おおまかに次のように働きます。

- シンボルは**現在のパッケージ**（名前空間）の文脈で解決される。
- パッケージをまたぐ名前は、**公開（export）されたものだけ**が外から見える。公開されていない名前を外から触るのはエラー。
- 解決は**遅延（実行時）**に行われる。これは、フォームの合間にパッケージの状態が変わっても、インタプリタと同じ意味を保つための方針です。

この章では実装の詳細には踏み込みません。ここで押さえたいのは、「**名前は文脈の中で解決される**」「**公開されたものだけが境界を越える**」という 2 点です。これはパッケージを使う／使わないにかかわらず、名前を設計するときの土台になります。

> より正確な解決規則は `docs/symbol-resolution.md` にまとまっています。

---

## 3. パッケージの位置づけ——ISLISP 規格外の拡張

ここが本章の肝です。`defpackage` / `in-package`（`:use` や `:export` を伴う、Common Lisp 流のパッケージ機能）は、**ISLISP 規格そのものには含まれていません**。isl では、これらを**拡張**として提供しています。

つまり、パッケージに依存したコードは「isl の拡張に依存したコード」であり、**規格準拠（strict）の観点からは移植性が保証されません**。3 周目の目的は規格準拠版の isl-calc を作ることなので、ここでは方針をはっきりさせます。

> **方針**: strict を意識する isl-calc では、`defpackage` / `in-package` に**依存しない**。名前空間は「命名規律」で確保する。

パッケージ機能そのものは強力で、大きなアプリケーションでは有用です。そこで本書は、パッケージのフルな活用を **4 周目（拡張編）** に回します。そこでは isl-calc を Web サービス化し、拡張機能を積極的に使う文脈で `defpackage` を導入します（[拡張編 2-1.1](../02-extended/01-extended-primitives.md) 以降）。

---

## 4. strict での名前設計——接頭辞という規律

パッケージを使わずに名前衝突を避ける、最も素朴で確実な方法が**接頭辞**です。isl-calc の公開 API に、モジュールを表す接頭辞 `calc-` を付けます。

```lisp
;; 公開 API（接頭辞 calc- で名前空間を表現）
(defun calc-parse (e) ...)        ; S式 → AST
(defgeneric calc-eval-node (node env))
(defun calc-eval (e) ...)         ; 入口：素朴に評価
(defun calc-try  (e) ...)         ; 入口：失敗を型付きで返す
```

一見すると素朴ですが、利点は本質的です。

- **衝突しにくい** — `calc-parse` は、他所の `parse` とぶつからない。
- **由来が読める** — 名前を見ただけで「これは calc モジュールのもの」とわかる。
- **移植性が保てる** — 拡張（パッケージ）に一切依存しないので、規格準拠の範囲に収まる。

AST ノードのクラス名（`num-node` など）も、規模が大きくなれば `calc-num-node` のように接頭辞を付ける設計が考えられます。要は「**公開する名前に、モジュールの出自を織り込む**」——これがパッケージ無しで名前空間を表現する規律です。次章の isl-calc v3 は、この規律に沿って公開 API を `calc-` で統一します。

---

## 5. 参考：パッケージを使う場合（拡張・予告）

拡張編での姿を、文法だけ予告しておきます。パッケージを使うと、接頭辞の代わりに「箱」で名前を仕切り、公開する名前だけを `:export` で外に出せます。

```lisp
;; ※ これは拡張機能。strict 移植コードでは使わない。
(defpackage :calc
  (:use :islisp)
  (:export :parse :eval :try))
(in-package :calc)
```

こうすると、箱の中では短い `parse` のまま書け、外からは `calc:parse` のように資格付きで呼べます。接頭辞規律を「言語機構」に格上げしたもの、と捉えると両者の関係が見えます。実際に動かすのは 4 周目です。

---

## ★ 処理系の中では（コラム）

isl はパッケージを、グローバル状態 `*package-registry*`（パッケージの表）と `*current-package*`（いま居るパッケージ）で管理し、シンボルは `resolve-symbol-in-package` で現在パッケージの文脈から解決されます。接頭辞規律は、この機構を使わずに「名前そのものにモジュール名を埋め込む」素朴なやり方で、処理系から見ればただのグローバルなシンボルです。だからこそ拡張に一切依存せず、strict でも確実に動きます。パッケージ機構の詳細は 6 周目（内部編）で扱います。

## 演習

1. **接頭辞への統一**: part1 の isl-calc v2 の公開関数（`parse` / `calc` など）を、`calc-` 接頭辞に一括で改名する方針を書き出せ。どの名前を公開 API とし、どれを内部専用とするか線引きせよ。
2. **衝突の再現（思考）**: `parse` という名前を 2 つの別モジュールが定義したら何が起きるか、シンボル解決の観点で説明せよ。接頭辞がなぜそれを防ぐかも述べよ。
3. **規格判定**: 次のうち strict で移植性が保証されるのはどれか——(a) `calc-parse` を `defun` で定義、(b) `(defpackage :calc ...)`、(c) `with-handler` でエラーを捕捉。理由とともに答えよ。

## まとめ

- プログラムが育つと**名前衝突**が問題になる。名前を仕切る箱が**名前空間**。
- シンボルは**文脈（現在パッケージ）の中で解決**され、**公開された名前だけ**が境界を越える。
- `defpackage` / `in-package` は Common Lisp 由来で、**ISLISP 規格外の拡張**。strict 移植コードでは**非依存**に保つ。
- 代わりに**接頭辞規律**（`calc-`）で名前空間を表現する。衝突しにくく、出自が読め、移植性を保てる。
- パッケージのフル活用は 4 周目（拡張編）で。
