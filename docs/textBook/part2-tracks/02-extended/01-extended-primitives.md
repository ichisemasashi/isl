# E1.1 拡張プリミティブの世界

> スパイラル **4 周目** / 拡張編（part2 第 2 編）第 1 章

3 周目（標準編）では、isl-calc から拡張を取り払い、ISLISP 規格の範囲だけで動く**strict な芯**を固めました。4 周目は、そこから方向を反転させます。今度は isl の**拡張機能を積極的に使い**、isl-calc を「履歴を残し、ネットワーク越しに使える実用ツール」へ育てます。

キーになる設計思想は——**「移植可能な芯の外側に、拡張の層を足す」**。3 周目で `calc-parse` / `calc-try` を規格準拠に保ったからこそ、ファイルやソケットといった拡張は「芯を汚さず外側に載せる層」として気持ちよく足せます。この章では、まず拡張機能の全体像を地図として眺めます。

## 学習目標

この章を終えると、次のことができるようになります。

- extended プロファイルで使える拡張機能の全体像をつかむ。
- 拡張機能と strict 規格の**境界**を意識できる。
- 「芯は規格準拠、外殻は拡張」という層構造の設計方針を理解する。

## 前提知識

- 標準編（[S1.1](../01-standard/01-strict-profile.md)〜[S1.4](../01-standard/04-calc-strict-edition.md)）を読了していること。

## 1. extended は「全部入り」の既定

思い出してください。isl はプロファイルを指定しなければ **extended（拡張）が既定**です。

```sh
./bin/isl examples/hello.lsp        # 既定で extended
./bin/isl --profile strict ...      # こちらが「削った」モード
```

つまり part1（0〜2 周目）で普通に isl を使っていたとき、私たちはずっと extended の上にいました。3 周目で初めて `--profile strict` を付け、拡張を外した世界を見たわけです。4 周目は既定の extended に戻り、今度はその**拡張を武器として使い倒します**。

---

## 2. 拡張機能の地図

isl の拡張は、おおまかに次のカテゴリに分かれます（詳細は `docs/compliance-matrix.md`）。

| カテゴリ | 代表的な関数 | この編での用途 |
|---------|------------|--------------|
| システム連携 | `getenv` / `setenv` / `system` / `debug` / `break` | （補助的に触れる程度） |
| **ファイル I/O** | `with-open-file` / `read-line` / `write` / `probe-file` | **履歴の永続化**（[E1.2](02-file-io-history.md)） |
| **ネットワーク（TCP）** | `tcp-listen` / `tcp-accept` / `tcp-send` / `tcp-receive-line` | **サーバの待ち受け**（[E1.3](03-networking.md)） |
| ネットワーク（TLS） | `tls-listen` / `tls-accept` / `tls-send` … | HTTPS 化（TCP とほぼ対称） |
| ネットワーク（UDP） | `udp-open` / `udp-send` / `udp-receive` … | （紹介のみ） |
| HTTP クライアント | `http-get` / `http-post` / `http-head` | （紹介のみ） |
| データベース | `sqlite-*` / `postgres-*` / `mysql-*` | （紹介のみ） |
| FFI | `ffi-call` / `load-foreign-library` | （紹介のみ） |

この編で主役になるのは、**ファイル I/O** と **TCP ネットワーク**です。この 2 つがあれば、「式を評価する Web サービス＋履歴永続化」という 4 周目のゴールに到達できます。TLS・UDP・HTTP クライアント・DB・FFI は、拡張の広さを知ってもらうための紹介にとどめます。

---

## 3. トレードオフ——移植性と実用性

3 周目で強調したとおり、拡張に頼るほど**移植性は下がります**。ファイル I/O やソケットは ISLISP 規格の外にあり、別の処理系では同じ関数が無いかもしれません。それでも拡張を使うのは、**実用性**が要るからです。履歴を残せない・ネットワークに出られない電卓は、ツールとしては物足りません。

大事なのは、この 2 つを**対立させず、層で分ける**ことです。

```
┌─────────────────────────────┐
│  HTTP 外殻（拡張: ソケット・I/O）   │  ← 4周目で足す層
├─────────────────────────────┤
│  評価コア（規格準拠: calc-parse/try）│  ← 3周目で固めた芯（移植可能）
└─────────────────────────────┘
```

芯は規格の範囲に保ち、拡張は外殻に隔離する。こうすれば、「評価ロジックは他処理系へ移植でき、外殻だけ差し替える」という選択肢が残ります。4 周目の isl-calc は、この層構造を体現します。

> **strict で拡張を呼ぶとどうなるか**: [S1.1](../01-standard/01-strict-profile.md) で見たとおり、strict では拡張プリミティブが登録されず `<undefined-function>` になります。つまり「うっかり芯に拡張を混ぜていないか」は、`--profile strict` で実行すれば検出できます。層の境界を守れているかの点検に使えます。

---

## ★ 処理系の中では（コラム）

拡張プリミティブも、規格のプリミティブと同じく `install-primitives!` の中で `(def 'name proc)` として環境に登録されます。違いは登録の**タイミング**だけで、strict プロファイルでは拡張分の登録がスキップされます（`disable-extended-primitives!` 相当）。だから「拡張が使えるかどうか」は、初期環境にその名前が載っているかどうか、という単純な話に帰着します。ネットワークやファイルの実体は、実装言語 Gauche のソケット/ポート機能を薄くラップしたものです。詳細は 6 周目（内部編）で。

## まとめ

- **extended が既定**。4 周目は拡張を積極活用し、isl-calc を実用ツールへ育てる周。
- 拡張は**システム連携・ファイル I/O・ネットワーク（TCP/TLS/UDP）・HTTP クライアント・DB・FFI** に大別される。この編の主役は**ファイル I/O と TCP**。
- 拡張は移植性を下げるが実用性を上げる。両者を**層で分離**する（規格準拠の芯＋拡張の外殻）のが設計の要。
- 芯に拡張が漏れていないかは `--profile strict` で点検できる。
