# echospace 音声・動画チャット要件定義 (v0.1)

この文書は `examples/echospace` に追加する音声・動画チャット機能の要件です。  
既存のテキストチャット要件は [REQUIREMENTS.md](/Volumes/SSD-PLU3/work/LISP/islisp/isl/examples/echospace/REQUIREMENTS.md) を正とし、本書はその上位拡張として扱います。

## 1. 目的

- `echospace` を Discord ライクな「テキスト + 音声 + 動画」コミュニケーションアプリへ拡張する
- ISL 製 Web アプリにおけるリアルタイムシグナリングとメディア通話の実装例を提供する
- 将来的な画面共有、録画、通知、モバイル対応へ拡張しやすい基盤を作る

## 2. スコープ

本機能では、最低限以下を対象とする。

- 音声チャンネル一覧表示
- 音声チャンネルへの参加 / 離脱
- マイク音声の送受信
- 参加者一覧のリアルタイム更新
- カメラの ON / OFF
- 参加者ごとの映像表示
- ミュート / ビデオ停止 / 自己ミュート状態表示

本フェーズでは以下を対象外とする。

- 画面共有
- 録画
- ステージチャンネル
- サーバー横断通話
- SIP / 電話連携
- ノイズ抑制や背景ぼかしの高度機能
- モバイルアプリ最適化

## 3. MVP の定義

音声・動画 MVP は以下を必須とする。

- 既存ワークスペース配下に音声チャンネルを定義できる
- 1 チャンネルに複数ユーザーが同時参加できる
- 参加ユーザー同士で音声通話できる
- カメラを有効化したユーザーの映像を表示できる
- ブラウザ上でミュート / ミュート解除、カメラ ON / OFF を切り替えられる
- 通話参加状態が他ユーザー画面に反映される
- 接続失敗や権限拒否時に UI 上で理由が分かる

MVP では以下を後回しにしてよい。

- 10 人超の大規模通話最適化
- サーバー録音
- 参加者の音量個別調整
- 発話検知の高度演出
- 低帯域自動最適化の細かな制御

## 4. 前提アーキテクチャ

音声・動画チャットは、単純なポーリングではなくリアルタイム双方向通信を前提とする。

- メディア伝送は `WebRTC` を第一候補とする
- サーバー側はシグナリングを担当する
- シグナリング経路は `WebSocket` を第一候補とする
- `SSE` は通話制御には不十分なため補助用途に限定する

MVP 時点の接続モデル:

- 小規模利用では `mesh` 接続を第一候補とする
- 1 チャンネルの推奨同時接続人数は `2..6` 人程度とする
- 将来の `SFU` 導入を阻害しないイベント設計とする

## 5. 機能要件

### 5.1 音声チャンネル

- 各ワークスペースはテキストチャンネルに加えて音声チャンネルを持てる
- 音声チャンネルは `name`, `slug`, `topic`, `display_order` を持つ
- ユーザーは所属ワークスペースの音声チャンネルにのみ参加できる
- 同時に参加できる音声チャンネルは 1 つまでとする

### 5.2 通話参加

- ユーザーは UI 上の「Join Voice」操作で音声チャンネルへ参加できる
- 参加時にマイク権限を要求する
- ユーザーは「Leave Voice」で通話を離脱できる
- 離脱時はメディアストリームと `PeerConnection` を確実に破棄する

### 5.3 音声通信

- 参加中ユーザーの音声は、同一チャンネル参加者へ配信される
- 自分自身の音声は原則ローカル再生しない
- ミュート時は送信トラックを無効化する
- UI 上にミュート状態を明示する

### 5.4 動画通信

- 参加中ユーザーは任意でカメラを有効化できる
- カメラ権限が許可された場合のみ映像送信を行う
- カメラ停止時は映像トラックを無効化または除去する
- 他ユーザー画面では、映像未送信時にプレースホルダを表示する

### 5.5 参加者状態

- 参加者一覧は以下の状態を持つ
  - `connected`
  - `muted`
  - `video_enabled`
  - `speaking` 任意
- 状態変化はリアルタイムに反映する
- 同一ユーザーの複数タブ参加は MVP では許可してもよいが、表示上区別できる設計が望ましい

### 5.6 接続制御

- 新規参加者が入室した際に既存参加者へ通知される
- 参加者間で `offer` / `answer` / `ICE candidate` を交換できる
- 切断や通信失敗時に UI を復旧可能な状態へ戻す
- ブラウザ再読込時は再接続を試みるか、少なくとも安全に再参加できる

### 5.7 テキストチャットとの連携

- 既存テキストチャネル UI と同じワークスペースナビゲーション内で音声チャンネルを見せる
- ユーザーが現在どの音声チャンネルに参加しているかを分かるようにする
- 将来的に「テキストチャンネルと音声チャンネルを同時表示」できる構造を維持する

## 6. UI 要件

音声・動画機能では以下の UI を持つ。

- 左カラムまたはチャンネル一覧内に音声チャンネル群を表示する
- メイン領域または下部固定バーに通話パネルを表示する
- 通話パネルに以下を含む
  - 参加 / 離脱ボタン
  - ミュート切替
  - カメラ切替
  - 現在参加中チャンネル名
  - 参加者数
- 動画有効時は参加者ごとの映像タイルを表示する

MVP の UI 方針:

- デスクトップ優先
- 参加者 6 人程度まではタイル表示で扱える
- モバイル最適化は任意

## 7. API / シグナリング要件

最低限以下の経路を持つ。

- `GET /echospace/workspaces/{workspace_slug}/voice/{channel_slug}`
  - 音声 / 動画 UI を含む HTML 表示
- `GET /echospace/api/workspaces/{workspace_slug}/voice/{channel_slug}/participants`
  - 現在参加者一覧取得
- `POST /echospace/api/workspaces/{workspace_slug}/voice/{channel_slug}/join`
  - 参加状態の作成
- `POST /echospace/api/workspaces/{workspace_slug}/voice/{channel_slug}/leave`
  - 参加状態の削除
- `WS /echospace/ws`
  - シグナリングイベント交換

WebSocket イベント種別の初期案:

- `voice.join`
- `voice.leave`
- `voice.state`
- `webrtc.offer`
- `webrtc.answer`
- `webrtc.ice_candidate`
- `webrtc.renegotiate`
- `webrtc.error`

## 8. データ要件

既存テーブルに加え、最低限以下の概念を扱う。

- `voice_channels`
  - `id`, `workspace_id`, `slug`, `name`, `topic`, `display_order`, `created_at`
- `voice_sessions`
  - `id`, `workspace_id`, `voice_channel_id`, `user_id`, `session_token`, `joined_at`, `last_seen_at`, `left_at`
- `voice_state`
  - `id`, `voice_session_id`, `muted`, `video_enabled`, `speaking`, `updated_at`

MVP での永続化方針:

- 通話の現在参加状態はメモリ常駐でもよい
- ただし将来の観測や復旧のため、少なくとも参加 / 離脱イベントは記録できる形が望ましい
- `voice_sessions` は履歴用途と現在値用途のどちらを兼ねてもよい

## 9. 権限要件

- 基本ロールは既存の `member` / `admin` を流用する
- `member` は所属ワークスペースの音声チャンネル参加を許可する
- `admin` は音声チャンネル作成 / 管理を許可する
- 非所属ワークスペースの音声チャンネルへ参加不可
- ログインしていないユーザーはシグナリング経路に接続不可

## 10. 非機能要件

### 10.1 性能

- MVP は小規模通話に最適化する
- 1 音声チャンネル推奨人数は `2..6`
- 参加人数増加時に性能劣化することを UI / ドキュメントで明示してよい

### 10.2 セキュリティ / プライバシー

- ブラウザ権限はユーザー操作時のみ要求する
- 明示的参加なしにマイク / カメラ送信を開始しない
- 未認証ユーザーへシグナリング情報を出さない
- WebSocket 接続にもセッション認証を適用する
- HTTPS 環境を前提とする
  - `getUserMedia` と実運用 WebRTC のため、ローカル開発以外では実質必須

### 10.3 可用性

- 通話失敗時にページ全体が壊れないこと
- 1 接続失敗が他参加者全体へ波及しないこと
- 切断時に UI がハングせず再参加できること

### 10.4 可観測性

- 参加 / 離脱
- シグナリング失敗
- `offer/answer` 失敗
- ICE 接続失敗
- 権限拒否

上記をログ出力対象とする。

## 11. 運用要件

- `examples/webserver` 上で動かせる構成を優先する
- ただし `WebSocket` 対応が必要なため、`webserver` 側の拡張が前提になる可能性がある
- STUN サーバー設定を環境変数で差し替え可能にする
- 将来的に TURN サーバー設定も外部注入できるようにする

想定設定:

- `ISL_ECHOSPACE_SIGNALING_MODE`
- `ISL_ECHOSPACE_WS_PATH`
- `ISL_ECHOSPACE_STUN_URLS`
- `ISL_ECHOSPACE_TURN_URLS`
- `ISL_ECHOSPACE_TURN_USERNAME`
- `ISL_ECHOSPACE_TURN_CREDENTIAL`

### Phase 1

- 音声チャンネルモデル追加

- 参加者状態 API

### Phase 2

- WebSocket シグナリング導入
- 音声のみ WebRTC 通話
- ミュート切替

### Phase 3

- カメラ映像
- 映像タイル UI
- 再接続改善

### Phase 4

- 画面共有
- 大人数対応
- SFU 検討
